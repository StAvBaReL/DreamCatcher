Here is the revised project breakdown, strictly divided and numbered for two developers. The tasks are separated to minimize merge conflicts, ensuring one focuses heavily on the user lifecycle and local data, while the other focuses on external integrations, the remote database, and the global feed.

Testing has been removed, and the tasks are directly aligned with your final project guidelines and mockups.

### Developer 1 (Stav): Identity, Profile, and Local Infrastructure

**Task 1: Project Foundation & Navigation Architecture**

* **What needs to be done:** Set up the mandatory Git repository for version control. Establish the base Android MVVM architecture. Implement the Navigation Graph (Nav Graph) and Fragments to handle all screen transitions defined in the mockup, ensuring `safeArgs` is used for passing parameters between screens.

**Task 2: User Authentication Flow**

* **What needs to be done:** Build the Login and Register screens according to the mockup. Integrate Firebase Authentication to handle user registration and login via email and password. You must implement an auto-login feature so returning users bypass the login screen, as well as a functional logout button.

**Task 3: User Profile Management**

* **What needs to be done:** Develop the Profile Screen UI. Implement the logic to display the current user's nickname and profile picture. Crucially, build the "Edit Profile" functionality so the user can update their display name and profile image. (User deletion is not required).

**Task 4: Local Database (Room) & Offline Caching**

* **What needs to be done:** Set up the local SQLite database using the Room library. You must implement custom caching for post objects. Use the Picasso library strictly for caching images. **Important guideline constraint:** You are forbidden from using Firebase's built-in offline persistence to satisfy the local storage requirement; it must be done manually via Room and Picasso.

**Task 5: "My Journal" / Personal Posts Screen**

* **What needs to be done:** Build a dedicated screen (as mandated by the requirements) where the logged-in user can view *only* the posts they have uploaded. This screen should heavily rely on the Room database you built in Task 4 to ensure the user can view their previously generated dreams even without an internet connection.

### Developer 2 (Rom): AI Integration, Remote Data, and Social Feed

**Task 1: External REST API Integration (Dream Generation)**

* **What needs to be done:** Build the Dream Creation Screen UI based on the mockup. Integrate an external text-to-image REST API (like OpenAI DALL·E 3 or Google Nano Banana). Take the user's text input, send the network request, handle loading states gracefully, and display the generated dream image on the screen.

**Task 2: Remote Database Setup (Firebase Firestore)**

* **What needs to be done:** Configure Firebase Firestore. Design the document structure for "Posts" so that each entry saves the text description, the image URL, a timestamp, and the author's user ID. Ensure the database rules allow for reading and writing based on authentication status.

**Task 3: Post Management (CRUD Operations)**

* **What needs to be done:** Connect the output of Task 1 to Task 2. When a user clicks "Post", upload the generated image and text to Firestore. Furthermore, implement the required post-management features: users must be able to edit an existing post (both the text and the image) and delete their own posts entirely.

**Task 4: Main Social Feed Screen**

* **What needs to be done:** Develop the Main Feed Screen UI to display a list of all dreams shared by all users on the platform. Map the data pulled from Firestore into a RecyclerView. This fulfills the core requirement that one user can upload content and another user can view it.

**Task 5: Incremental Loading (Pagination)**

* **What needs to be done:** Implement incremental loading (טעינה מדורגת) for the Main Social Feed. Instead of pulling every post from Firestore at once, fetch them in chunks (e.g., 10 at a time) as the user scrolls down the feed. This ensures the app remains performant and adheres to the project guidelines.
